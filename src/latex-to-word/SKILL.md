# 作用说明

本文件是 `latex-to-word` skill 的总控说明书。

当用户提出以下任务时，应优先使用本 skill：
- 将 LaTeX 工程转换为 Word（`.docx`）
- 希望尽量保留图片、表格、标题、公式、参考文献、目录和交叉引用
- 需要一条可重复执行的“预检查 → 转换 → 审核 → 人工修正清单”流水线
- 需要在 Windows 11 本地工程目录中完成转换

本 skill 的目标不是“神奇地无损一键转换任意 LaTeX 工程”，而是：
1. 用 `Pandoc` 完成主体转换；
2. 用本 skill 中的脚本完成预检查、规范化、后检查和人工修正清单生成；
3. 将人工修正严格限制在高风险点，不做整篇返工。

---

# 适用范围

本 skill 适用于以下输入：
- 有明确主文件的 LaTeX 工程，例如 `main.tex`
- 多文件工程（含 `\input` / `\include`）
- 含图片资源、参考文献库（`.bib`）、常见数学环境、常见图表环境的文档
- 目标输出为 Word 文档 `.docx`

本 skill 不直接负责：
- 将输出美化成出版级排版成品
- 重新设计文档结构
- 改写原文内容
- 替用户决定参考文献样式之外的学术规范问题
- 为了“看起来更像 PDF”而牺牲 Word 的可编辑性

---

# 核心原则

## 1. 保结构优先于保视觉
优先保证以下内容正确保留：
- 标题层级
- 图题、表题
- 参考文献
- 公式语义
- 编号体系
- 内部引用的可恢复结构

不要为了接近 PDF 外观而把大量结构压扁成普通文本或整页图片。

## 2. 保 Word 可编辑性优先于静态复刻
输出文档必须优先满足以下要求：
- 正文仍可编辑
- 公式尽量是 Word 原生数学对象
- 目录可更新
- 图表标题可识别
- 交叉引用尽量可恢复

## 3. 不覆盖原始工程
除非用户明确要求，否则不得直接覆盖原始 `.tex`、图片、`.bib` 或其他源文件。
所有规范化处理应优先写入工作副本或中间文件。

## 4. 不擅自增加流程
本 skill 只允许使用当前目录中已经定义的以下组件：
- `scripts/precheck.py`
- `scripts/normalize_tex.py`
- `scripts/convert_with_pandoc.py`
- `scripts/convert_with_pandoc.sh`
- `scripts/postcheck_docx.py`
- `scripts/build_manual_fix_list.py`
- `templates/reference.docx`
- `rules/supported_envs.md`
- `rules/downgrade_policy.md`
- `rules/acceptance_criteria.md`

未经用户要求，不新增额外转换阶段、不引入额外外部系统、不改用 PDF-to-Word 作为主路线。

## 5. 明确区分“自动完成”和“人工收口”
对于交叉引用、复杂表格、子图、算法、代码块等高风险对象：
- 先尽可能自动处理；
- 自动处理失败时，必须明确记录；
- 最终通过人工修复清单收口；
- 不允许静默失败。

# 标准执行流程

本 skill 的标准流程固定为五步，禁止跳步。

## 第一步：预检查（`precheck.py`）

### 目标
在转换前检查工程是否具备基本可转换条件，并尽早暴露高风险项。

### 必须检查的内容
- 主文件是否存在
- `\input` / `\include` 链是否闭合
- 图片资源是否缺失
- `.bib` 是否存在且可访问
- 是否存在明显未定义的 `\ref` / `\eqref` / `\cite`
- 是否存在高风险环境：
  - 复杂表格
  - TikZ
  - 子图
  - 算法环境
  - 代码环境
  - 自定义环境
- 是否存在可能阻断 Pandoc 的明显结构问题

### 行为要求
- 预检查阶段只读，不改文件。
- 预检查必须生成结构化报告。
- 对于“可继续但有风险”的问题，标记为告警。
- 对于“无法继续转换”的问题，标记为阻塞错误。

### 输出要求
预检查结果必须至少区分：
- PASS：可直接进入下一步
- WARN：可继续，但需要重点复核
- FAIL：必须先修复，不能进入主转换

---

## 第二步：规范化（`normalize_tex.py`）

### 目标
在不改变原始语义的前提下，将 LaTeX 工程整理成更适合 Pandoc 处理的形式。

### 允许进行的规范化
- 统一路径表示
- 统一 `\caption` / `\label` 的常规顺序
- 统一常见交叉引用写法
- 展开少量简单、无歧义的自定义宏
- 将已知无法直接转换但允许降级的对象替换为规范化占位形式
- 为后续人工修复对象插入明确标记或记录

### 明确禁止的行为
- 改写正文语义
- 删除用户内容
- 大规模重排结构
- 擅自替换参考文献条目内容
- 覆盖原始 `.tex` 文件

### 输出要求
- 规范化后的工作副本
- 规范化报告
- 所有降级动作都必须可追踪

---

## 第三步：Pandoc 主转换（`convert_with_pandoc.py` / `convert_with_pandoc.sh`）

### 目标
调用 Pandoc 完成从 LaTeX 到 `.docx` 的主体转换。

### 必须保证
- 使用 `templates/reference.docx` 作为 Word 样式模板
- 正确设置资源路径，确保图片可以被 Pandoc 找到
- 正确处理参考文献输入
- 正确生成 Word 文档
- 优先保留标题、图片、表格、参考文献和数学对象

### 行为要求
- 本步骤只负责主体转换，不承担全部修复责任。
- 若发生转换错误，必须保留命令日志和错误摘要。
- 若存在可转换但不完全保真的对象，不应因追求完美而中止全流程，除非规则文件将其列为阻塞项。
- Win11 默认入口应使用：`python scripts/convert_with_pandoc.py --work-root <path>`
- Linux/macOS 可使用：`bash scripts/convert_with_pandoc.sh --work-root <path>`

### 输出要求
- 生成 `.docx`
- 生成转换日志或转换摘要
- 明确记录本步骤中的告警和降级项

---

## 第四步：docx 后检查（`postcheck_docx.py`）

### 目标
对生成的 Word 文档做结构级检查，确认主体转换结果是否满足最低可交付条件。

### 必须检查的内容
- `.docx` 文件是否存在且可打开
- 图片是否已嵌入
- 表格是否存在
- 图题、表题是否保留
- 标题层级是否存在
- 公式是否尽量为 Word 数学对象
- 参考文献是否已生成
- 是否存在明显断链或空对象
- 是否存在需要人工重建的关键交叉引用

### 行为要求
- 本步骤只检查结果，不直接修改 `.docx` 主体内容。
- 所有问题都必须进入结构化报告。
- 需要人工修复的项，必须给出明确类别，而不是含糊描述为“格式问题”。

---

## 第五步：人工修复清单生成（`build_manual_fix_list.py`）

### 目标
将预检查、规范化、主转换和后检查的结果合并，生成一份面向用户的人工修复清单。

### 清单必须包含
- 建议修复顺序
- 每个待修对象的位置描述
- 问题类别
- 是否影响最终交付
- 推荐修复方式

### 建议修复顺序
1. 更新 Word 全文字段
2. 检查目录
3. 检查图目录、表目录
4. 重建关键公式引用
5. 重建关键图表引用
6. 修复复杂表格
7. 修复子图、算法、代码块
8. 终审参考文献与编号

### 输出要求
清单必须是可直接交给用户执行的文档，不能只输出内部调试信息。

# 各脚本职责说明

本节用于约束每个脚本的边界，防止重复实现和职责漂移。

## `scripts/precheck.py`
职责：
- 只负责“转换前检查”
- 不修改源文件
- 识别阻塞项与高风险项
- 输出结构化预检查报告

不负责：
- 规范化
- 实际转换
- docx 检查
- 人工修复清单汇总

## `scripts/normalize_tex.py`
职责：
- 只负责“生成适合 Pandoc 的规范化工作副本”
- 不覆盖原始工程
- 输出规范化报告

不负责：
- 直接调用 Pandoc
- 检查 `.docx`
- 生成最终人工修复清单

## `scripts/convert_with_pandoc.sh`
职责：
- 只负责调用 Pandoc 完成主转换
- 接收明确输入，产出明确 `.docx`
- 输出转换摘要

不负责：
- 预检查规则判断
- 规范化策略设计
- docx 结果审校

## `scripts/convert_with_pandoc.py`
职责：
- 只负责调用 Pandoc 完成主转换（Windows 11 默认入口）
- 接收明确输入，产出明确 `.docx`
- 输出转换摘要

不负责：
- 预检查规则判断
- 规范化策略设计
- docx 结果审校

## `scripts/postcheck_docx.py`
职责：
- 只负责检查生成结果
- 不回写源工程
- 不修改原始 `.docx`
- 输出结构化后检查报告

不负责：
- 二次转换
- 自动大规模修复 docx
- 重新解释 LaTeX 语义

## `scripts/build_manual_fix_list.py`
职责：
- 聚合多阶段报告
- 输出面向用户的修复清单

不负责：
- 再次执行转换
- 替用户直接修 Word
- 修改原始工程或 docx

# 降级策略执行要求

遇到无法完整保真的对象时，必须遵循 `rules/downgrade_policy.md`，并满足以下原则：

## 可接受的降级
- 极复杂 TikZ 图可先转为图片
- 极复杂表格可进入人工修复清单
- 少量自定义环境可降级为普通段落
- 个别难以自动恢复的交叉引用可转为“可定位、可人工重建”的状态

## 不可接受的降级
- 整篇公式全部转图片
- 删除图题或表题
- 丢失参考文献
- 丢失图片对象
- 将全部交叉引用静默替换成普通文本且不报告
- 删除用户正文内容而不记录

每次降级都必须：
- 被记录
- 可追踪
- 可向用户解释
- 可纳入人工修复清单

# 交叉引用处理原则

本 skill 必须特别关注以下三类引用：

1. `\cite`
2. `\ref`
3. `\eqref`

处理目标如下：

## `\cite`
优先目标：
- 保留文内引用
- 保留文末参考文献
- 尽量保留可点击跳转

## `\ref` / `\eqref`
优先目标：
- 保留锚点关系或可恢复结构
- 在自动无法完全恢复为 Word 原生交叉引用时，必须：
  - 标记位置
  - 记录原始指向
  - 加入人工修复清单

不得将内部引用问题静默忽略。

# 结果汇报格式要求

每次执行本 skill 后，最终汇报必须包含以下部分：

## 1. 执行概况
- 工程根目录
- 主文件
- 输出文件
- 是否成功生成 docx

## 2. 预检查摘要
- 阻塞错误数量
- 告警数量
- 主要高风险对象类别

## 3. 规范化摘要
- 是否生成工作副本
- 做了哪些规范化
- 是否有降级项

## 4. 转换摘要
- Pandoc 是否执行成功
- 图片、表格、参考文献、公式的总体状态
- 是否有明显转换失败对象

## 5. 后检查摘要
- 目录是否可生成/可更新
- 图题、表题是否保留
- 交叉引用是否存在明显断链
- 公式是否符合要求

## 6. 人工修复建议
- 最优先处理的 3～10 个问题
- 修复顺序
- 是否达到“可交付初稿”标准

# 实现纪律

执行本 skill 时必须遵守以下实现纪律：

1. 所有脚本都应优先输出清晰、结构化、可追踪的信息。
2. 所有报错都应说明：
   - 什么失败了
   - 为什么失败
   - 失败影响范围
   - 下一步怎么做
3. 不得用模糊描述替代具体结果，例如：
   - 不要只说“有些图片有问题”
   - 应说“缺失 2 张图片：xxx、yyy”
4. 不得把“未知是否成功”表述成“已经成功”。
5. 不得为追求自动化而绕开验收规则。
6. 不得因为局部对象失败而隐瞒整体风险。

# 完成判定

仅当以下条件满足时，才可以将本次任务标记为“已完成”：

1. 预检查已完成；
2. 规范化已完成；
3. Pandoc 主转换已完成；
4. 后检查已完成；
5. 人工修复清单已生成；
6. 用户拿到至少一个 `.docx` 与两份文本报告；
7. 所有阻塞项都已显式处理或显式报告。

若 `.docx` 未生成，则本次任务只能标记为“失败并已输出诊断结果”，不能标记为“完成”。
